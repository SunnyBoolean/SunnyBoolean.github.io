java中一些重要的关键字

发表于 2018-10-31 | 分类于 Java基础

       
java中的关键字(保留字)接近50个，这里我们列出比较难懂但是又很关键常用的一些关键字：

final、finally、static、synchronized、transient、volatile

final

       关于final这个关键字，从字面意思就是永久的，固定不变的，在java中的使用率几乎100%。

使用场景

final主要用于修饰变量、方法、类，但是不能修饰代码块。

      final修饰的方法是不能被子类重写的。所以final就可以起到方法被别人重写的可能。

      final修饰类是无法被子类继承的。所以final就可以起到类被继承的可能，比如String类就被定义为final类型，所以我们无法重写String类。

总结

  1. 全局变量如果不在定义的时候就初始化编译器会提示报错，而局部变量则不用在定义的时候初始化，他可以在后面赋值一次。
  2. 全局变量和局部变量都只能赋值一次。
  3. final修饰变量后，改变量的值不能再被更改，如果是引用类型的变量，则改变量保存的是对象的内存地址。

static

      想了半天，这个关键字不知从何说起了，因为它实在不是几段话能概括得了的。我们还是从使用场景说起吧。

使用场景

static 主要修饰全局变量、方法、类、代码块。不能修饰局部变量和构造器

修饰全局变量

      static修饰全局变量后，就意味着这个变量和类的对象无关了，此变量只跟类有关，而且只会被加载一次，当他被加载到内存后，后面就不会再次加载初始化，但是你可以随意对其进行赋值。static变量在初始化的时候是伴随这类的加载和初始化的。当我们new一个对象的时候类会被加载初始化、同时static变量也会被初始化加载。当我们直接通过类名调用static变量的时候，同时也伴随着类的加载和初始化。

修饰方法

      static修饰方法跟修饰全局变量有相同的作用，即都是由其类名直接调用，且不需要创建对象(当然也可以通过对象来调用)，同理的，如果类没有被初始化，直接使用类名调用方法的话也会使类被加载。

       一般情况下的用法我们就不多讲了，这里说一些比较特殊的情况下的用法

static类型的同步方法

      如果一个方法既有static又有synchronized关键字修饰，那么我们就称这个方法为静态同步方法，这种同步方法会让这个类都获取一把锁，即在同一时刻只能有一个线程执行这个方法，如果其他线程要使用则需要排队等待

      这种静态的同步方法和非静态同步方法有何不同呢？非静态同步方法就是没有static的同步方法，这种方法虽然也会获得锁，但是他获得的是对象锁，简单来说就是多个线程里同一时刻同一个对象只能有一个线程能执行此对象的方法，其它线程需要让这个对象的方法排队等待。相反，多个线程如果是不同的对象调用方法，那么就无需上锁排队等待了，毕竟方法是属于对象的，这也不难理解。

synchronized

      要理解synchronized就需要知道Java里的多线程的知识。因为在多线程环境下，我们需要保证任务同步执行，这时候就需要用到synchronized和volatile关键字。且在多线程环境下我们需要保证一个由多线程共享的对象或者方法在同一时刻只能由一个线程任务来使用。

      实际上，Java中通过synchronized来保持线程同步和操作系统中互斥内存块的概念一样，在Java中的Object类型中，都是带有一个内存锁的，在某个线程获取改内存锁后，其它线程无法访问该内存，从而实现Java中简单的同步、互斥操作。

      也就是说，如果是synchronized（this）的话，其内存锁是针对相同的对象的互斥，也就是多个线程使用同一个对象会出现互斥现象，但是多个线程使用不同的对象去调用方法则不会互斥，因为他们每个对象的内存区域不相同。相同的，如果是synchronized(static)，则多个线程不管使用类的哪个对象都会互斥，因为static关键字注定了方法或者代码块的内存是所有对象共有的。

      所以记住，如果不想某些基本类型和String字段被序列化，你加了transient关键字就不要加final，否则他依然会被序列化。同时也不要加static关键字。总之，尽量final和static类型的变量会一直被jvm序列化，无法阻止(final的引用类型变量除外).

HashMap序列化原理

       正常情况下一个类的HashMap成员变量是可以被序列化的，HashMap源码是实现了Serializable接口的，但是他存储元素的数组table确实transient，也就是table是不可以被序列化的，那么Hashmap是如何保证序列化呢？看源码发现HashMap有readObject和writeObject来自己实现序列化，而不使用虚拟机的序列化。Serializable接口说得很清楚，如果类实现了Serializable接口且重写了writeObject()和readObject()方法，那就意味着自己实现序列化算法，不再使用jvm的序列化算法。通常我们自己定义类的时候是直接实现Serializable接口即可，至于序列化则交由虚拟机实现，只不过HashMap是自己实现序列化了，他这样做是因为其存储元素是考key的hashCode来存储和查找的，序列化和反序列化的key值的hashCode可能会不一致，导致元素错乱。所以HashMap自己实现了一套序列化机制，且把table和key值都定义为transient类型的就是避免被jvm序列化。

序列化总结

（1）transient可以阻止成员变量被序列化。
（2）所有非transient的无论是private、public、default的变量都会被序列化。
（3）常量表达式的成员变量会被jvm默认序列化，此时transient失效。
（4）HashMap是transient的最佳实践。

volatile

      volatile关键字的使用场景是多线程环境。他的主要作用是保证变量在多线程中变量的原子性、可见性。再此之前我们先了解一下内存模型：

      （1）原子性：一个操作或者多个操作，要么全部执行并且执行的过程不会被任何因素打断，要么就都不执行。的还好，如果是异步的就会出现严重后果。

      （2）可见性：是指当多个线程访问同一个变量时，一个线程修改了这个变量的值，其他线程能够立即看得到修改的值。在多线程情况下，这种特性尤其重要。

      （3）有序性：即程序执行的顺序按照代码的先后顺序执行。这涉及到指令重排序一般来说，代码在编译后，处理器在执行相应的命令时会对指令进行重排序导致程序并不是顺序执行，但是最终结果和顺序执行的结果是一致的。指令重排序只是处理器为了更高效和高效率的处理事情而做出的优化。虽然在单线程下指令重排序不会影响最终结果，但是如果是多线程情况下，多个线程在执行相同的代码，顺序不同，则会出现结果不一样的情况。

      volatitle的作用就是有效的保证了程序的有序性和可见性。他能阻止变量写到缓存，而直接写到主存。
